## Goal

Create five separate, fully mocked desktop-first frontend prototypes under `designs/1` through `designs/5` so the product direction can be reviewed in-browser before refactoring the production frontend.

## Constraints

- Each prototype must be self-contained and runnable independently with Bun.
- Use React, Vite, TypeScript, and Tailwind.
- No real backend integration or live data access.
- No tests are required for these review-only prototypes.
- Do not abstract shared code across the five designs for this pass.
- The prototypes should feel intentional and varied, not generic AI-generated dashboards.

## Shared Product Model

The information architecture stays consistent across all five designs so the review compares visual language and UX treatment instead of comparing different product concepts.

### Domain

- Users can belong to multiple bands.
- Each band can contain multiple projects.
- Each project contains songs and project-scoped activity.
- Songs can contain generated stems and collaborator-uploaded stems.
- Timeline comments belong to users and remain visible in history after resolution.

### Core Flow

1. Landing page that explains collaborative song practice, review, stems, tabs, and notes.
2. Fake authentication flow for sign in, register, and join/create a band.
3. Band and project selection entry point.
4. Project dashboard with recent activity since last login.
5. Song library with upload entry points and song states.
6. Song detail page with metadata, collaborators, comments, stems, actions, and processing status.
7. Player page with transport, chord progression, tab display area, fretboard, timeline notes, and stem controls.

### Collaboration Model

- Comments are multi-user and visible on song screens and the playback timeline.
- Timeline note markers show author context, including avatar previews on hover.
- Comments support open and resolved states.
- Resolved comments remain visible in history and filter views.
- Project activity shows what changed since the user last logged in.
- Notification surfaces stay scoped to a project to avoid cross-band noise.

### Stem Model

- Stems may be system-generated or user-uploaded.
- Each stem shows uploader attribution.
- System-generated stems use `System` attribution.
- Each stem supports a short description.
- Stem descriptions are editable only by the uploader in the mocked UX.
- Instrument groups can contain multiple non-archived versions.
- Users can enable or disable stems during playback.
- Users can switch which active version of an instrument is used during playback.
- Users can archive old versions while keeping non-archived variants available for comparison.
- Stems should be downloadable for use in external DAWs.

### State Model

The prototypes should visibly represent asynchronous states because the real system already includes long-running processing and will become more queue-driven over time.

Required example statuses:

- uploaded
- processing
- ready
- failed
- needs review
- generated by system
- uploaded by collaborator

## Five Design Directions

### 1. IBM-Inspired Product System

- Structured enterprise grid
- Strong hierarchy and disciplined spacing
- Neutral surfaces with restrained accent usage
- Emphasis on clarity, reliability, and project context

### 2. DAW-Inspired Studio Console

- Inspired by audio workstations such as GarageBand and similar studio tools
- Track-lane thinking, darker control surfaces, tactile playback treatment
- Stem switching and transport become the visual center of gravity

### 3. Editorial Modern Workspace

- Large typography, stronger asymmetry, more expressive storytelling on landing
- Clean collaboration surfaces and high-content readability
- Less “tool chrome,” more confident content framing

### 4. Scandinavian Instrument Lab

- Calm, light visual system with natural tones and reduced chrome
- Gentle density, highly legible panels, quiet confidence
- Focus on band collaboration without making the tool feel corporate

### 5. Live Performance Control Surface

- Bolder, stage-inspired identity with high contrast and sharper emphasis
- More kinetic and expressive than the other directions
- Still clean enough to support dense playback and stem data

## UX Principles

- Keep the prototypes desktop-first.
- Make complex playback data progressively disclosed instead of stacking everything at once.
- Treat collaboration as a first-class feature, not an overlay.
- Keep metadata readable and lightweight.
- Use activity, badges, drawers, tabs, or side panels only when they reduce clutter.
- Preserve comparability by keeping the underlying product model and routes consistent across all five prototypes.

## Routes To Mock In Every Prototype

- `/` landing
- `/auth` sign in/register
- `/bands` band and project switcher
- `/projects/:projectId` project dashboard
- `/projects/:projectId/songs` song library
- `/projects/:projectId/songs/:songId` song detail
- `/projects/:projectId/songs/:songId/player` player view

## Implementation Notes

- Each prototype should ship with hard-coded mock data tailored to the shared domain.
- Each prototype should expose the same general route structure but with distinct layouts and styling.
- Each prototype should be runnable independently, ideally on port `3001` when started from its own directory.
- The designs directory is intentionally disposable review scaffolding and should not be coupled to the production `frontend/` app.
