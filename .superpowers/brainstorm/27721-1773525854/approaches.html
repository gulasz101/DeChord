Implementation Approach — How should stacking & animation work?

All three produce the same visual result. The difference is in complexity and smoothness.

A

Pure CSS + JS class toggling Recommended

Toasts are a flex-col container fixed to top-left. Entry uses a CSS keyframe (slideInLeft), exit adds a fading-out class triggering a CSS fade. Author color is derived from a hash of the author name mapped to one of 8 preset gradients. Zero new dependencies.

Pros

  • No new deps
  • Predictable, testable
  • Works with existing Tailwind setup

Cons

  • No layout animation when toasts reorder
B

Framer Motion AnimatePresence

Each toast wrapped in a Framer Motion <motion.div>. AnimatePresence handles mount/unmount transitions automatically. The stack smoothly reflows when the oldest toast exits — no jumping. Already used elsewhere in the project or easy to add.

Pros

  • Buttery layout transitions on reorder
  • Declarative, easy to tweak

Cons

  • Adds ~40 KB dep if not already present
C

CSS @starting-style + transition (modern native)

Uses the new CSS @starting-style rule (Chrome 117+, Firefox 129+) for true enter transitions on newly inserted DOM nodes. Exit via JS opacity transition. Most declarative pure-CSS solution, no JS animation logic at all.

Pros

  • Zero JS animation code
  • Future-proof native approach

Cons

  • Browser support still maturing
  • No reorder animation
